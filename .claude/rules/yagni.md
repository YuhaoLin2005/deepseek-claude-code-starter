# YAGNI — 懒人高效法则

> 融合自 [DietrichGebert/ponytail](https://github.com/DietrichGebert/ponytail)（46K+ stars）的核心哲学：**最好的代码是没写过的代码。**
> 本规则是 ponytail 在本配置体系中的适配版本——保留"懒"的决策框架，同时与安全/测试/代码质量规则协调共存。

## 决策阶梯（6-Rung Ladder）

每次准备写代码前，从第1级开始，找到第一个适用的就停：

| 级 | 问题 | 行动 |
|----|------|------|
| **1** | 这真的需要存在吗？ | 推测性需求 → 跳过，一行说明原因（YAGNI） |
| **2** | 标准库能做吗？ | 用标准库 |
| **3** | 平台原生能力覆盖吗？ | CSS 替代 JS，`<input type="date">` 替代日期选择器库，DB约束替代应用层校验 |
| **4** | 已安装的依赖能解决吗？ | 用它。绝不为几行代码能做的事加新依赖 |
| **5** | 能一行搞定吗？ | 一行 |
| **6** | **只有到这一步：** | 写刚好能工作的最小代码 |

> 阶梯是条件反射，不是研究项目。两阶都适用 → 取更高的那阶，继续前进。第一个能用的懒方案就是正确方案。

## 行为准则

- **不主动抽象**：没有只有一个实现的接口，没有只生产一个产品的工厂，没有不改变的配置项。
- **不写"以后"用的脚手架**：以后可以自己搭。
- **删除优于添加，无聊优于聪明**：聪明是凌晨3点有人要破译的东西。
- **最少文件，最短 diff 获胜**。
- **复杂请求？** 先交付最懒版本，同一句话质疑需求："做了 X；其实 Y 已经覆盖了。确定要完整的 X 吗？"
- **两个标准库方案一样大？** 选边界情况正确的那个。懒是少写代码，不是选更脆弱的算法。

## `ponytail:` 注释约定

每一个刻意推迟的简化，用以下格式标记，让技术债**可见、可收割**：

```
// ponytail: <当前简化方案>; upgrade to <升级路径> when <触发条件>
```

示例：
```java
// ponytail: global lock; upgrade to per-account lock when throughput > 1000 req/s
```
```sql
-- ponytail: SELECT * for now; upgrade to explicit column list when table > 20 cols
```
```js
// ponytail: O(n²) scan for small N; upgrade to Map lookup when items > 100
```

简单读作意图，而非无知。不写注释的简化才是地雷。

## 输出风格

- **代码先行**，解释在后（最多三行）：跳过了什么，什么情况下需要补。
- 解释比代码长？删掉解释——每一段为简化辩护的段落，就是伪装成散文偷渡回来的复杂度。
- **例外**：用户明确要求的报告、实验文档、PR 摘要、分阶段说明 → 完整输出（这不是偷渡，是需求）。

模式：`[代码] → 跳过: X，何时补: Y。`

## 绝不偷懒的地方

以下**永远不能简化**（与 `security.md`、`code-quality.md` 一致）：

- 信任边界的输入校验
- 防止数据丢失的错误处理
- 安全措施（认证、授权、加密）
- 无障碍基础
- 用户**明确要求**的功能（YAGNI 只适用于**推测性/推断性**需求）
- 硬件校准参数（物理世界不是理想模型——时钟会漂移、传感器会偏移）

## 测试：分级适用

与 `testing.md` 协调——不取消 80% 覆盖率，而是分级：

| 代码类型 | 测试要求 |
|----------|---------|
| 用户面向功能、API端点、关键业务逻辑 | **80%+ 覆盖率**，按 `testing.md` 的 TDD 流程 |
| 内部工具函数、配置映射、简单工具类 | **至少一个可运行检查**：`assert` 自测 或 一个最小测试文件 |
| 纯一行代码、getter/setter、常量定义 | 可跳过（YAGNI 也适用于测试） |

> 非平凡逻辑（有分支、循环、解析、涉及钱/安全/数据的路径）必须留下至少一个会失败的可运行检查。无框架、无 fixture、无 per-function 套件，除非明确要求。

## 依赖原则

```
标准库 > 已安装依赖 > 领域标准库 > 小众依赖
```

- **Spring Security、MyBatis-Plus、JWT、Hutool、Lombok** 等是领域标准骨架，不是"新依赖"。
- "不新增依赖"主要针对：能用 3 行代码替代的工具库、未经验证的小众包、功能重复的依赖。

## 冲突裁决

当本规则与现有规则表面冲突时：

| 本规则说 | 现有规则说 | 裁决 |
|----------|-----------|------|
| 不写抽象 | DRY：提取重复逻辑 | 等待真实重复 **≥3 次**，不投机提取 |
| 不新增依赖 | 优先知名库 | stdlib 先行；领域标准库（密码、JWT、ORM）不受此限 |
| 一行搞定 | 函数 <50行、清晰命名 | 一行可读性够就行；**清晰 > 简短** |
| 最少解释 | 详细 PR/文档 | 协作场景（PR、实验报告）完整输出 |
| 质疑需求 | 用户请求 | **只质疑推测性/未明确的需求**，用户明确需求是神圣的 |
